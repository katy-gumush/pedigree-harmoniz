QA & Automation Engineer Home Assignment 
“Pedigree Explorer – Quality & Automation” Dear candidate, we are excited to see your work. 
Here is the assignment description. 
Goal 
Build an automated test suite and supporting data-manipulation code that validate the correctness of a dogs pedigree system, focusing specifically on: 
● Pedigree integrity in the data 
● Pedigree correctness and robustness at the API level 
● Pedigree display in the UI 
You may assume there is a working “Dogs Pedigree” application (UI + API) that loads the dataset and exposes pedigree information. 
You do not need to build or submit the application itself. Your deliverable is the tests and data-manipulation code only. 
There is no time limit. 
We care about clarity, completeness, good engineering thinking, and especially the quality of your: 
● Tests 
● Handling of expected errors 
● Thought process 
Dataset
Use the dogs pedigree dataset in the CSV file: 
● Dogs Pedigree.csv 
You may treat this as the “clean” baseline data and also create modified versions of it for error simulation. 
Application Under Test (Context) 
The system you are testing should support, at a minimum: 
● A list/table of dogs 
● A dog card (“birth certificate”) with fields such as: 
○ ID 
○ Name 
○ Breed 
○ Sex 
○ Height 
○ Weight 
● A pedigree view for a selected dog, showing ancestors and descendants (up to 5 generations each side) 
The exact tech stack, deployment method, and visual design are not important for this assignment. 
Your focus is on verifying that the pedigree is correct and behaves correctly under both normal and error conditions. 
Focus of the Assignment 
The core of the assignment is to: 
● Understand what can go wrong with pedigree data and display
● Manipulate the data to simulate at least one concrete pedigree-related error ● Write automated tests that detect those issues at: 
○ Data level 
○ API level 
○ UI level 
You do not need to cover every aspect of the system. 
Concentrate on meaningful pedigree-related checks. 
Testing Requirements – Overview 
For each of the three layers (data, API, UI), we expect: 
● A short explanation of your test strategy and assumptions (in comments or a separate text/markdown file) 
● At least one meaningful automated test focused on pedigree integrity or pedigree display 
● At least one test (overall) that uses deliberately corrupted data to demonstrate how failures are detected 
The three layers are described below. 
Data-Level Pedigree Tests 
Write tests that run directly on the dataset (for example via a script or notebook) and validate pedigree integrity. 
Example rules (choose a sensible subset) 
● Required fields are present for each record (for example: id, name, breed, sex) ● id is unique across all records 
● sire_id and dam_id are either empty or reference valid existing ids in the dataset
● A dog is not its own parent (no self-parenting) 
● No simple cycles in the immediate pedigree (for example A is parent of B and B is parent of A) 
● Optional sanity checks such as height and weight being non-negative or within reasonable bounds 
What we expect you to do 
● Use Dogs Pedigree.csv as a “clean” baseline 
● Create at least one manipulated version of the dataset with a deliberate pedigree issue (for example a broken parent ID, duplicated ids, or a small parent-child loop) 
● Show, through your tests, that the corrupted data causes a clear and understandable test failure 
You may implement these tests in any language or framework you like, for example: ● Python with pytest 
● A Jupyter notebook with assertions 
API-Level Pedigree Tests 
Assuming the application exposes API endpoints or a queryable interface, write automated tests that verify pedigree correctness at the API boundary. 
Example checks (adapt to your actual API) 
● A search or list endpoint (for example a “dogs list” or “dogs search” call) returns structured data with expected fields and types 
● A dog details endpoint (for example “get dog by id”) returns consistent information for a known dog id
● A pedigree endpoint (for example “get pedigree for dog id”) returns: 
○ Ancestors and descendants that match the relationships in the dataset ○ No duplicate dogs in the pedigree for a single root dog 
○ Pedigree depth limited to the expected number of generations (for example up to 5) 
What we expect you to include 
At least one pedigree-focused API test that: 
● Confirms the pedigree returned by the API for a known dog matches what your data-level logic expects 
● Demonstrates how the system behaves when there is an inconsistency (for example a missing parent id in the data), if you can simulate this scenario 
If there is no traditional REST API, you can treat any stable boundary (for example a function, query layer, or RPC call) as the “API” for the purposes of testing. 
UI-Level Pedigree Tests 
Write at least one automated test that drives the UI and verifies that the pedigree is displayed correctly for a selected dog. 
Typical flow to cover 
● Open the main view and display the dogs list/table 
● Locate a known dog (for example by searching or filtering) 
● Open that dog’s card 
● Open the pedigree view and verify that: 
○ The correct root dog is shown 
○ At least some ancestors and/or descendants are displayed as expected (structure or key fields)
You do not need to test all visual details. Focus on: 
● Whether the UI shows a pedigree that is consistent with the underlying data or API 
● Whether the UI handles missing or problematic data gracefully (for example not crashing, and showing a reasonable fallback if applicable) 
You may implement UI tests using Playwright, Cypress, Selenium, or any similar tool. Data Manipulation and Error Simulation 
A key part of the assignment is demonstrating that you: 
● Anticipate realistic pedigree-related errors 
● Manipulate the data to create at least one such error scenario 
● Use your tests to detect and explain that error 
Example error scenarios 
● A dog whose sire_id or dam_id points to a non-existent id 
● Duplicate dog ids 
● A simple immediate loop (dog A is listed as parent of dog B, and dog B is listed as parent of dog A) 
● A dog that appears multiple times in its own ancestor tree in the pedigree view 
Include comments or a short note describing: 
● Which error scenario(s) you chose
